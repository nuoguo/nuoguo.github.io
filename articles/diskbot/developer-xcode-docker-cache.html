开发者的"隐形存储杀手"

如果你是 Mac 上的 iOS/macOS 开发者，或者经常用 Docker 跑容器，你的硬盘可能正在被悄悄吞噬。

DerivedData — 每次 Xcode 编译都会生成，一个项目动辄几百 MB，积累下来轻松超过 10GB。

模拟器缓存 — iOS 模拟器的数据、快照、日志，长期不清理能占 20GB+。

Docker 镜像和容器 — 拉过的镜像、停止的容器、悬空层，每个都是存储黑洞。

问题是：这些文件夹藏在 ~/Library 深处，系统存储管理根本看不到具体内容。手动找？路径太深，命令太复杂，还怕误删。

今天教你用 DiskBot 可视化扫描 + 一键清理，安全高效地找回磁盘空间。

准备工作

开始前，确保你已安装 DiskBot：

  • 系统要求：macOS 14 (Sonoma) 及以上
  • 下载方式：从官网或 GitHub Releases 获取 DMG 安装包

安装后首次打开，DiskBot 会请求"完全磁盘访问"权限——这是扫描 ~/Library 等系统目录的必要条件。

第 1 步：扫描磁盘，定位缓存目录

打开 DiskBot，点击左侧工具栏的 "扫描" 按钮，选择你的主目录（通常是 /Users/你的用户名）。

扫描完成后，你会看到一个 TreeMap 矩形树图：每个矩形代表一个文件夹，面积越大，占用空间越多。

▲ DiskBot TreeMap 视图 — 一眼看清哪些目录占用最多

重点关注这几个大块：

  • ~/Library/Developer/Xcode/DerivedData — Xcode 编译缓存
  • ~/Library/Developer/CoreSimulator — iOS 模拟器数据
  • ~/.docker — Docker Desktop 数据目录

如果看到这些目录的矩形特别大，恭喜你，清理空间的机会来了。

第 2 步：使用缓存检测功能

TreeMap 能帮你定位大目录，但 DiskBot 还有一个更强大的功能：智能缓存检测。

点击左侧工具栏的 "缓存" 图标，DiskBot 会自动扫描 16+ 种常见开发者缓存，并按安全等级分类：

关键优势： DiskBot 知道哪些缓存可以安全删除，哪些需要谨慎处理。比如 Xcode Archives 包含你的打包历史，删除前最好确认是否需要保留。

第 3 步：选择并清理缓存

在缓存列表中，勾选你想清理的项目。建议优先处理：

  1. DerivedData — 安全清理，下次编译会自动重新生成
  2. iOS 模拟器缓存 — 不影响现有项目，清理后模拟器会重新下载数据
  3. Docker 悬空镜像 — 未被任何容器使用的镜像层，清理无副作用

勾选完成后，点击右下角的 "移至废纸篓" 按钮。

DiskBot 会批量将选中的文件移到废纸篓，整个过程在后台异步执行，不会卡住界面。

预期结果： 清理完成后，你会在界面顶部看到释放的空间总量。实测一台开发机，清理 DerivedData + 模拟器缓存 + Docker 悬空镜像，轻松释放 32GB。

第 4 步：验证清理效果

清理后，建议做两件事确认一切正常：

1. 检查磁盘空间

点击菜单栏的苹果图标 → "关于本机" → "存储"，确认可用空间确实增加了。

2. 验证开发环境

打开 Xcode，编译一个项目。首次编译会比平时慢一点（因为 DerivedData 需要重新生成），但之后就会恢复正常速度。

如果使用 Docker，运行 docker images 确认常用镜像还在，悬空镜像已被清理。

常见问题

Q: 清理 DerivedData 会影响我的项目代码吗？

A: 不会。DerivedData 只包含编译产物和索引缓存，项目源代码完全不受影响。下次编译会自动重新生成。

Q: 清理 iOS 模拟器缓存后，模拟器还能用吗？

A: 可以。清理的是模拟器的数据缓存，不影响模拟器本身。下次启动模拟器时，系统会重新下载必要的数据。

Q: Docker 镜像被误删怎么办？

A: DiskBot 只清理"悬空镜像"（dangling images），即没有被任何容器使用的镜像层。你正在使用的镜像不会被删除。如果不小心删错了，用 docker pull 重新拉取即可。

Q: 多久清理一次比较合适？

A: 建议每月一次。如果项目多、编译频繁，可以每两周一次。DiskBot 的扫描很快，养成习惯只需几分钟。
